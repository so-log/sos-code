# 지속 가능한 코드

나는 아직 개발자로 일한지 2년이 채 안된 주니어 개발자이다. `The RED : 프론트엔드 Back to the Basics : 지속 가능한 코드작성과 성능 향상법 by 김태곤`을 들으면서 소름돋았던 적이 한 두번이 아니다. 강사님이 얘기해주시는 말들이 그동안 일하면서 했던 경험이나 느꼈던 생각과 일치했기 때문이다.
이 강의를 통해 앞으로 프론트엔드 개발자로써 개발을 할때 무엇에 중점을 두고 어떤 것들을 경계해야하는지 더욱 더 확신을 가지게 됐다.
이 글은 그렇게 정의된 나만의 개발 가치관이자 김태곤 강사님의 지속 가능한 코드 작성법이다.

## 좋은 코드

1. 간결하고 의도가 명확해야한다.
1. 주석 < 코드 만으로 의도파악이 가능해야 한다.
1. 단순하고 의존성이 적은 코드를 사용해야 한다.
   _2016 left pad 사건_ - [11줄의 코드, 인터넷을 패닉에 빠뜨리다](https://www.bloter.net/news/articleView.html?idxno=22900)
1. 하나의 함수, 메소드, 클래스는 하나의 의미만 담고 있어야 한다. (반복문이 많아지면 복잡도가 증가)

## 디버깅

증상이 아닌 원인을 수정해야한다.

### bug - 프론트엔드 관점에서 알아보기

1. 재현과정
   사용자의 액션이 있어야 발생하는 버그가 대부분이고 버그가 발생하는 플랫폼은 사용자의 환경이다. 따라서 정확한 재현 과정이 필요하다.
1. 왜 에러라고 생각하는 가?
   정확한 기대 동작이 있어야 유의미한 버그로 존재할수 있다.
   ex 로딩이 5초 이상으로 느린 것 -> 버그인가? -> '로딩시간이 3초 이상 지나면 안된다.' 라는 기대 동작이 있으면 이것은 버그이다.
1. 에러 메세지
   에러 메세지가 나타나는 문제는 해석만 잘하면 된다.

이처럼 버그를 만나지 않으려면 테스트가 중요하다.

## 테스트

### 좋은 테스트?

시스템에 관해 필요한 지식을 빠짐없이 전달할 수 있어야 한다.
단순히 어떤 결과를 내기 위해 실행된 코드의 양을 측정하는 **테스트 커버리지**가 꼭 중요한 것은 아니다.

- 공식 문서가 이해가 안간다면 테스트 코드를 봐라 -> 의도 파악이 가능하다, 엣지 케이스를 알 수 있다.
- **엣지 케이스(edge case)** 주어진 조건이 극한이 상황일 때 발생하는 문제를 말한다. 즉, 일부러 테스트가 실패할 값을 넣어서 테스트를 진행하는 것이다.

### 좋은 테스트의 조건

1. 실행 속도가 빨라야 한다.
1. 입출력 위주를 중심으로 작성해야 한다.
   비공개 메소드도 테스트를 해야하는가? X -> 모든 코드를 테스트하는게 목적이 아니라 공용의 인터페이스가 작동하도록 하는게 중점이여야 한다.
1. 버그를 찾을 수 있어야 한다, 테스트 시나리오를 잘 설정해야 한다.
1. 테스트 결과에 일관성이 있어야 한다.
1. 명확한 의도가 보여야 한다.

## TDD vs BDD

테스트를 먼저 작성하고 그 후 실제 코드를 작성하는 방법이다. BDD가 사용자 시나리오에 조금 더 집중하는 면이 있지만 비슷하다.

### 장점

1. 테스트를 먼저 작성하기 때문에 테스트 커비리지의 비율이 높아진다.
1. 테스트 되는 것만 코드로 작성하므로 코드가 방대해지지 않는다.
1. 버그로 발생하는 시간 낭비를 줄여준다.

### 방법

1. 테스트를 먼저 작성한다.
1. 테스트를 통과하는 코드를 작성한다.
1. 리팩터링

## 리팩터링

> 소프트웨어의 겉보기 동작은 그대로 유지한 채, 코드를 이해하고 수정하기 쉽도록 내부 구조를 변경하는 기법 <br />
>
> 마틴 파울러, 리팩터링 2판, 한빛미디어

비슷한 버그가 3번 이상 발견되면 리팩터링을 해야한다. <br />
리팩터링을 하기 전에 최소한의 안전장치인 TDD 테스트 코드를 작성하자.

### 방법

1. 함수 추출 : 분리한다.
1. 중간 변수 도입, 함수로 만들기 : 함수이름에 의도를 담아서 사용한다.
1. var < let < const
1. 가능한 순수 함수로 작성 : 신뢰성을 높이기 위해 사용한다.
1. 반복문 보다는 파이프 : 메서드를 사용한다.
   for -> filter, forEach
1. switch 대신 object literal
1. ES6 템플릿 문자열을 사용한다.
